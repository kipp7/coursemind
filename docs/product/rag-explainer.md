# RAG 是什么，为什么 CourseMind 先做 RAG

RAG 是 Retrieval-Augmented Generation，意思是“检索增强生成”。

在 CourseMind 里，它不是训练一个新模型，而是让系统先去查当前课程资料，再把查到的内容交给模型组织回答。

```text
学生问题
  -> 选择课程空间
  -> 检索课程大纲、课件、讲义、教师资料
  -> 找到相关片段和来源
  -> 模型基于这些材料回答
  -> 展示引用
  -> 教师审核和审计留痕
```

## 它解决什么问题

大模型本身不知道学校这门课的最新课件、老师要求、评分标准和课程安排。直接问模型，模型可能答得像真的，但不一定符合课程资料。

RAG 的价值是：

- 回答能基于当前课程资料，而不是只靠模型记忆。
- 可以展示引用，方便学生和老师追溯来源。
- 课程资料变化时，只需要更新知识库，不需要重新训练模型。
- 教师审核可以围绕具体引用和具体回答进行。
- 后续接 Dify、RAGFlow、私有向量库时，业务 API 不需要重写。

## RAG 和微调的区别

RAG 解决“现在这门课有哪些资料、答案依据在哪里”。

微调解决“模型长期应该如何表达、如何按老师标准评分、如何遵守学校规范”。

所以 CourseMind 的顺序是：

1. 先做 RAG，保证回答能引用课程材料。
2. 再做教师审核，收集老师确认和修正的数据。
3. 最后才考虑微调，把稳定的老师风格、评分标准和学校规范固化进模型行为。

## 当前 MVP 的状态

当前 MVP 已有 RAG gateway 边界，但默认还是 mock retrieval。也就是说，系统结构已经按 RAG 设计，真实资料解析、切片、索引和检索还在后续阶段。
