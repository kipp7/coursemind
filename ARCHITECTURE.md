# CourseMind 校园课程智能体架构草案

## 目标

先交付一个学校可演示、老师能理解、学生能试用的 Web MVP，再逐步升级成多课程、多角色、可私有化部署的课程智能体平台。

## MVP 边界

- Web 端课程智能体工作台
- 学生端：课程问答、复习提纲、术语解释、练习题生成
- 教师端：课程资料上传、FAQ 维护、回答风格与禁答范围配置
- 管理端：课程空间、模型供应商、知识库同步状态、审核状态
- 回答必须带引用来源，避免变成无依据的聊天机器人

## 推荐技术路线

```text
Web 前端
  -> 应用服务层
  -> RAG / Agent 平台
  -> 模型服务
  -> 课程资料库
```

### Web 前端

- 第一版可用静态原型快速确认交互和业务范围
- 正式版建议使用 React / Next.js
- 需要支持学校品牌、移动端、课程空间、角色权限

### 应用服务层

- 用户、角色、课程权限
- 对话记录与学习记录
- 文件上传与资料审核
- 调用 Dify / RAGFlow / 自研 RAG API
- 统一日志、限流、敏感内容拦截

### RAG / Agent 平台

优先接入开源平台，而不是第一天自研全部能力。

- Dify：适合快速搭建智能体应用、工作流和后台配置
- RAGFlow：适合复杂 PDF、课件、教材的文档理解
- AnythingLLM：适合小规模私有知识库工作区
- LangGraph / LlamaIndex：适合后期自研可控 Agent 流程

### 模型服务

- MVP：OpenAI-compatible API，便于切换供应商
- 中文私有化：Qwen / DeepSeek / GLM 等开源或开放权重模型
- 部署：vLLM / Ollama / 云厂商 API

## 为什么先 RAG，不先微调

课程内容会经常变化，微调不适合承载最新课件、作业说明、考试范围。第一阶段应使用 RAG，让模型从课程资料中检索并带引用回答。

微调适合第二阶段以后：

- 学习老师偏好的讲解风格
- 固化学校的答题规范
- 训练评分 rubric
- 优化特定学科的表达方式

## 数据模型雏形

- School：学校
- Department：院系
- Course：课程
- User：用户
- Role：学生 / 教师 / 管理员
- Document：课程资料
- Chunk：知识片段
- Conversation：会话
- Citation：引用来源
- ReviewTask：资料审核任务
- Rubric：评分或回答规范

## 关键风险

- 幻觉：必须要求引用和资料范围约束
- 权限：学生不应访问教师私密资料
- 版权：教材和课件需要明确授权范围
- 隐私：学生提问和学习记录要做脱敏与权限控制
- 教学责任：智能体回答应标注为辅助，不替代教师最终解释
- 成本：需要限流、缓存、模型分级调用

## 推荐迭代顺序

1. Web MVP 原型
2. 接 Dify 或 RAGFlow，跑通单门课程
3. 支持资料上传、解析、分块、引用
4. 增加教师审核和禁答规则
5. 增加统一登录与课程权限
6. 增加学习分析和教师看板
7. 基于真实问答数据做微调或提示词优化
